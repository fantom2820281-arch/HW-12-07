## Задание 1.

1.1 Опишите основные преимущества использования масштабирования методами:

- активный master-сервер и пассивный репликационный slave-сервер;
- master-сервер и несколько slave-серверов;
- дайте ответ в свободной форме.

### Решение

► РЕЖИМ 1: ОДИН ACTIVE MASTER + ОДИН PASSIVE SLAVE

Преимущества:
  1. Отказоустойчивость (failover):
     - Если master упал → slave можно быстро повысить до master
     - Минимальное время простоя (RTO ~ минуты)
  
  2. Бэкапы без остановки:
     - Бэкап делается на slave → master не нагружается
     - Пользователи не замечают процесса резервного копирования
  
  3. Разделение нагрузки чтение/запись:
     - Запись → только на master
     - Чтение (отчёты, аналитика) → на slave
     - Master не тормозит из-за тяжёлых SELECT-запросов
  
  4. Тестирование:
     - На slave можно тестировать миграции БД
     - Проверка новых версий приложения без риска для продакшена

► РЕЖИМ 2: ОДИН MASTER + НЕСКОЛЬКО SLAVE

Дополнительно к преимуществам выше:
  1. Горизонтальное масштабирование ЧТЕНИЯ:
     - Нагрузка чтения распределяется между N слейвами
     - Каждый новый slave = +100% к пропускной способности чтения
  
  2. Географическое распределение:
     - Slave в Европе, slave в Азии → низкая задержка для локальных пользователей
     - Master в одном дата-центре, slave в других регионах
  
  3. Специализация слейвов:
     - Slave1 → для веб-приложения (OLTP)
     - Slave2 → для аналитики (OLAP, тяжёлые запросы)
     - Slave3 → для бэкапов
  
  4. Высокая доступность:
     - Падение одного slave не влияет на систему
     - Master продолжает работать, остальные slave подхватывают

► СРАВНИТЕЛЬНАЯ ТАБЛИЦА

|Критерий          -   | 1 master + 1 slave | 1 master + N slave  |
|:--------------------:|:------------------:|:-------------------:|
|Отказоустойчивость    | Базовая            | Высокая             |
|Масштабирование чтения| Низкое             | Линейное            |
|Стоимость             | Низкая             | Средняя/высокая     |
|Сложность настройки   | Простая            | Средняя             |
|Гео-распределение     | Нет                | Да                  |

### Решение

2.1

► ИСХОДНЫЕ ДАННЫЕ

Таблицы:
  - users (id, name, email, region, created_at)
  - books (id, title, author, isbn, price, shop_id)
  - shops (id, name, city, address)

► ВЕРТИКАЛЬНЫЙ ШАРДИНГ (Partition by Table)

Принцип: каждая таблица → на отдельный сервер

Схема размещения:
 
-[Вертикальный шардинг](./img/01-2_1-12.7.drawio.pdf)

Преимущества:
  - Простота: каждая БД маленькая, быстрая
  - Изоляция нагрузки: тяжёлые запросы к books не влияют на users
  - Независимое масштабирование: можно добавить ресурсы только для books

Недостатки:
  - JOIN между таблицами невозможен напрямую → нужна денормализация
  - Транзакции между серверами сложны (2PC / Saga)

► ГОРИЗОНТАЛЬНЫЙ ШАРДИНГ (Partition by Row)

Принцип: строки одной таблицы распределяются по серверам

Режимы работы серверов:
  - USERS MASTER    → запись+чтение (OLTP)
  - USERS SLAVE-1   → чтение для веб-приложения
  - USERS SLAVE-2   → аналитика + бэкапы
  - BOOKS MASTER    → запись+чтение
  - BOOKS SLAVE-1   → чтение, шард A (id 1..1M)
  - BOOKS SLAVE-2   → чтение, шард B (id 1M+)
  - SHOPS MASTER    → запись+чтение
  - SHOPS SLAVE-1   → чтение для API

#### ПРИНЦИПЫ ПОСТРОЕНИЯ

1. Вертикальный шардинг — для изоляции доменов:
   - users, books, shops — разные предметные области
   - У каждой своя нагрузка и требования
   
2. Горизонтальный шардинг — для книг (самая большая таблица):
   - По id: простой, предсказуемый
   - По жанру/автору: семантический, но сложнее балансировать
   
3. Репликация — для отказоустойчивости и чтения:
   - Master-slave на каждом вертикальном шарде
   - Async репликация (достаточно для большинства случаев)

4. Маршрутизация — на уровне приложения:
   - ORM знает, на какой сервер идти
   - Или отдельный Proxy (ProxySQL, Vitess)

# Создаем учебную отказоустойчивую инфраструктуру в Yandex-Cloud

### Расположение серверов по зонам

Единая точка входа

```bash

                    ┌─────────────────┐
                    │   APPLICATION   │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │   ProxySQL /    │  ← Единая точка входа
                    │   HAProxy       │     (порт 6033)
                    └────────┬────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
   ┌────▼────┐          ┌────▼────┐          ┌────▼────┐
   │  USERS  │          │  BOOKS  │          │  SHOPS  │
   │ CLUSTER │          │ CLUSTER │          │ CLUSTER │
   └────┬────┘          └────┬────┘          └────┬────┘
        │                    │                    │
   ┌────┴────┐          ┌────┴────┐          ┌────┴────┐
   │ Master  │          │ Master  │          │ Master  │
   │ Slave-1 │          │ Slave-1 │          │ Slave-1 │
   │ Slave-2 │          │ Slave-2 │          │ Slave-2 │
   └─────────┘          └─────────┘          └─────────┘
```

[2] РЕГИОНЫ (ГЕО-РАСПРЕДЕЛЕНИЕ) — ДЛЯ ГЛОБАЛЬНЫХ ПРИЛОЖЕНИЙ

► Если приложение для всего мира:
   - Регион "Москва" → пользователи из РФ/Европы
   - Регион "Владивосток" → пользователи из Азии
   - Регион "Франкфурт" → пользователи из ЕС

► В каждом регионе — СВОЙ полноценный кластер (master + 2 slave)
► Между регионами — асинхронная репликация (master → remote master)

► Переключение при катастрофе:
   - Global Load Balancer (Cloudflare, AWS Route53, Yandex DNS)
   - При падении Москвы → DNS переключает на Владивосток
   - Время переключения: 30 сек — 5 мин (зависит от TTL DNS)

[3] ГДЕ ХРАНИТЬ БЭКАПЫ (ПРАВИЛЬНО)

► Правило 3-2-1:
   - 3 копии данных
   - 2 разных носителя
   - 1 копия В ДРУГОМ ГЕО-РЕГИОНЕ

► Схема:

```bash
   ┌─────────────────────────────────────────────────────────┐
   │  Регион Москва (AZ-1, AZ-2, AZ-3)                       │
   │  ├─ Master + 2 Slave (горячие данные)                   │
   │  └─ Бэкапы → каждый час в Object Storage (S3)          │
   └─────────────────────────────────────────────────────────┘
                          │
                          │ асинхронная репликация
                          ▼
   ┌─────────────────────────────────────────────────────────┐
   │  Регион Владивосток (DR - Disaster Recovery)            │
   │  ├─ Read-only slave (полная копия данных)               │
   │  └─ Object Storage с бэкапами из Москвы                 │
   └─────────────────────────────────────────────────────────┘
```

► Бэкап-сервер в том же регионе — ПЛОХО (при падении региона потеряешь всё)
► Бэкап в другом облаке — ХОРОШО (защита от отказа провайдера)

```bash


   ┌─────────────────────────────────────────────────────────┐
   │  Kubernetes Cluster (приложения)                        │
   │  ├─ Frontend (3 пода)                                   │
   │  ├─ Backend API (5 подов)                               │
   │  └─ Workers (10 подов)                                  │
   └─────────────────────────────────────────────────────────┘
                          │
                          │ подключение по IP
                          ▼
   ┌─────────────────────────────────────────────────────────┐
   │  Отдельные ВМ (БД)                                      │
   │  ├─ MySQL Master (8 vCPU, 32 GB RAM, NVMe SSD)          │
   │  ├─ MySQL Slave-1 (в другой AZ)                         │
   │  └─ MySQL Slave-2 (в третьей AZ)                        │
   └─────────────────────────────────────────────────────────┘
```
► Альтернатива: Managed Database (Yandex Managed MySQL, AWS RDS)
   - Провайдер сам делает репликацию, бэкапы, failover
   - Ты платишь за удобство
   - Для 90% проектов — ЛУЧШИЙ ВЫБОР

[5] КАК ПРАВИЛЬНО ОРГАНИЗОВАТЬ (ИТОГОВАЯ СХЕМА)

Для production-приложения среднего размера:

```bash 

┌─────────────────────────────────────────────────────────┐
│  РЕГИОН "МОСКВА" (3 зоны доступности)                    │
│                                                          │
│  Kubernetes Cluster:                                     │
│  ├─ Frontend, Backend, Workers                           │
│  └─ Подключается к БД по внутреннему IP                  │
│                                                          │
│  БД (отдельные ВМ):                                      │
│  ├─ AZ-1: MySQL Master (8 CPU, 32 GB, NVMe)              │
│  ├─ AZ-2: MySQL Slave (синхронная репликация)            │
│  ├─ AZ-3: MySQL Slave (асинхронная)                      │
│  └─ ProxySQL (2 шт, для балансировки чтения)             │
│                                                          │
│  Бэкапы:                                                 │
│  └─ Каждый час → Yandex Object Storage (S3)              │
└─────────────────────────────────────────────────────────┘
                          │
                          │ асинхронная репликация
                          ▼
┌─────────────────────────────────────────────────────────┐
│  РЕГИОН "ВЛАДИВОСТОК" (Disaster Recovery)                │
│  ├─ MySQL Read-Only Slave (полная копия)                 │
│  ├─ Kubernetes Cluster (для аварийного запуска)          │
│  └─ Object Storage (копия бэкапов)                       │
└─────────────────────────────────────────────────────────┘
                          │
                          │ DNS failover
                          ▼
              Global Load Balancer
              (Cloudflare / Route53)
```

► "Как правильно организовать?"
   - Managed MySQL (проще) ИЛИ отдельные ВМ (дешевле, гибче)
   - 3 зоны доступности в одном регионе
   - Реплика в другом регионе
   - Бэкапы в Object Storage + в другом регионе
   - Global DNS для переключения
